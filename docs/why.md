# Why this exists

Every other document here explains *how* the method works — the lanes, the
handoffs, the ledger, the cycle. This one explains *why* it exists, and the
honest answer is more personal than any theory of context economics. It started
with not knowing what I was doing.

## Not knowing which model to use

When I started, I reached for the most capable model for everything — not
because the work needed it, but because I couldn't tell which work did. It
wasn't obvious which tasks actually required the expensive tier and which a much
cheaper one would have handled just as well, so the safe-feeling default was to
use the strongest model available for all of it, trivial and hard alike.

That default is easy to fall into and expensive to keep. It got more expensive
when I ran up against the limits of what I was paying for and had to notice how
much of that budget was going to work that never needed it. Most of what a
project needs day to day — formatting, drafting from a settled outline, the
mechanical passes — doesn't need the most powerful model; it needs the cheapest
one that will get it right. I didn't have a way to tell those apart. The method
is, at bottom, the skill I was missing, written down: a way to know what to
spend where.

## We were never taught to tier

The subscription era taught a habit of not thinking about it. Pay once, use
freely, don't count — and for most things that was fine. But capability is
metered now in a way a flat monthly fee hides, and most of us never learned to
match the tool to the task. We reach for the biggest model the way you'd leave
every light in the house on: not because you need them all, but because
switching them off is one more thing to think about.

Choosing deliberately — this task needs judgment, that one needs a checklist,
this one just needs doing — is a skill, and it's one almost nobody was taught.
Tokenomics is an attempt to write that skill down, so it can be used on purpose
instead of picked up by accident.

## Compute isn't free

There's a reason this matters past the monthly bill. Compute has a real cost —
economic, and environmental. Running a large model is not weightless: it draws
power, water, and hardware whether or not the task called for that much of them.
Spending the most expensive tier on work a smaller one could do isn't only
overpaying — it's waste, in the plain sense of the word.

So the discipline has a cleaner motive than saving money. Reserve the expensive
tier for the work that genuinely needs it, and use the smallest capable tier for
everything else. Better economics and less waste point the same way here, which
is part of why the method is worth its small overhead.

## What the discipline is

Everything else in this repository is the practical form of one idea: match the
model to the work.

- Route by the cost of getting it wrong, so the expensive tier is spent only
  where a mistake is actually expensive.
- Route down as far as your gates reach — keep judgment-priced capability for
  the judgment that has no cheap way to be checked.
- Write decisions down, so nothing that cost the expensive tier to reach has to
  be paid for twice.

None of it is about spending less for its own sake. It's about spending well —
putting the capability where it counts, and not burning it where it doesn't.
That the method came out of my own not-knowing, at the point where I was
overspending on nearly everything, isn't incidental to it. It's the reason it
exists.
